# Veilwright site work

## 1. Resolve the domain to a site — every time, don't assume

Before calling any `page.*`/`media.*`/`job.*`/`audit.*` tool, call `site_list` and match the domain the user mentioned (strip `https://`, `http://`, `www.`; compare case-insensitively) against each returned site's `url`.

- No match: tell the user this site isn't registered yet and that they need to run `site_add` first (WP-admin → Veilwright AI → create an API key, then "add my site: <url>, key ID ..., secret ...").
- More than one plausible match: ask which one before doing anything.
- Never fall back to whichever site a previous `site_select` left active in this process. Claude Desktop connects to Veilwright's MCP server once at app startup and that one process is plausibly shared by every open conversation — a different, unrelated chat may have already changed the "selected" site. Always resolve the domain explicitly.

## 2. Pass `siteId` explicitly on every call for this request

Once resolved, pass that site's `id` as the `siteId` argument on every tool call for this task (`page_create`, `page_createFromHtml`, `page_update`, `media_upload`, `job_status`, ...). Don't rely on `site_select` alone — it's a convenience for a single ad-hoc call, not something to lean on across a multi-step task.

## 3. Pick the right page tool

**Hard rule, not a preference — do not re-derive or re-ask this per task:** if Elementor (free and/or Pro) is active on the target site (check `site_healthCheck`'s `adapters`), the page gets built with **native Elementor widgets** — period. `page_createFromHtml`/`page_addElementorWidget`/`elementorTemplate_create` all produce real widgets and are all fine to use; a raw `html` widget dump is not, except as a last-resort for content with genuinely no matching native widget (see section 5's rule on this). If some other page-builder plugin were active instead of Elementor (this plugin currently has no adapter for one — check `site_healthCheck`'s `adapters` first), that builder's own mechanism would be the equivalent choice, not raw HTML. Only when **no page-builder plugin at all** is active does building in plain HTML/content become acceptable — and even then, ask the user how they'd like it done rather than assuming.

**`page_createFromHtml` can fail silently — never trust the job status alone, always verify with `page_getElementorData` afterward.** Confirmed real bug: an HTML source containing a `<style>` tag (even an empty/unused one) makes the job report `status: "completed"`, `result: { ok: true }` — a completely normal-looking success — while the page's actual `_elementor_data` is `elements: []`, totally empty. There is no `"failed"` status, no error message, nothing to catch this except reading the page's real persisted content back. **After every `page_createFromHtml` job reaches `"completed"`, call `page_getElementorData` and confirm `elements` is non-empty and roughly matches what was expected before telling the user the page is done.**

Separately, even when the converter *does* produce elements, its CSS fidelity is limited — confirmed real gaps: background colors/images, gradients, `border-radius`, `box-shadow`, and button styling (an `<a>` styled as a button becomes a plain `text-editor` widget, not a `button` widget) are not reliably carried over; only basic layout properties (`flex_direction`, `flex_gap`, `padding`) survive consistently. Treat `page_createFromHtml` as a **text/structure import**, not a pixel-accurate design import — for anything with real visual design (colors, buttons, cards, multi-column layouts), the hard rule above still applies: build it **widget-by-widget with `page_addElementorWidget`** (or `elementorTemplate_create` for a Theme Builder template) using explicit native Elementor settings (see `ELEMENTOR_WIDGETS.md`), not by hoping the HTML conversion carries the design over.

Whichever way a `page_createFromHtml` failure (silent or otherwise) shows up, the hard rule above still applies: rebuild widget-by-widget with `page_addElementorWidget`, matching the layout/content from the original HTML/design as closely as possible. Do **not** offer the user a choice between "fix the HTML and retry the converter" (e.g. stripping the `<style>` tag, inlining styles) and "rebuild with native widgets" — the native-widgets path is the only correct one on an Elementor site and should be taken directly, not surfaced as an equally-weighted option. Tell the user afterward what happened and why (the converter's limitation) rather than asking upfront which approach to use. If unsure which Elementor elements/widgets are enabled on the target site, the live Element Manager is the ground truth: `<site-url>/wp-admin/admin.php?page=elementor-element-manager`.

- **Plain content, no design/layout described** ("create an Impressum page with this text") → `page_create` (title + content).
- **A design, layout, or HTML is given or implied** ("build this page to look like...", HTML pasted in, a screenshot/description of a layout) → `page_createFromHtml`. This builds a *native* Elementor page (real widgets), not a content dump.
  - It's async: it returns `{ jobId, status: "queued", postId }` immediately. Poll `job_status(jobId)` — wait a couple seconds between checks — until the status is `"completed"` or `"failed"`. Don't tell the user the page is done before that.
  - On `"failed"`, report the actual error to the user, then rebuild with `page_addElementorWidget` per the hard rule above — don't silently retry, don't fall back to `page_create` (that would drop the design/layout entirely), and don't ask the user to pick a workaround strategy.
  - **Even a plain text/legal page (Impressum, AGB, Datenschutz, ...) still needs actual layout thought — don't just emit a flat stream of top-level `<h2>`/`<p>` siblings.** The converter maps every top-level element to its own container by default, so a flat list of 15 siblings produces 15 separate containers with default (often too loose, or inconsistent) spacing between every single one — exactly the "unnecessary container sprawl" a real user flagged after having to manually rebuild a page for this reason. Instead:
    - **Group each heading with the paragraph(s) that belong to it inside a wrapping `<div>` (or `<section>`)** — the parser already maps `div`/`section` to a single container holding its children, so this alone turns "15 top-level containers" into "one container per logical section, each with its own heading+text inside." This is the single highest-value change.
    - **Merge a block of directly-related lines into one `<p>` with `<br>` between them** (e.g. a name+street+city+country block) rather than one `<p>` per line — `<br>` is preserved as a real line break now, not silently dropped. Reserve separate `<p>` tags for genuinely distinct paragraphs/thoughts, not just visual line breaks within one block.
    - **Consider side-by-side grouping for naturally parallel short blocks** (e.g. contact info next to a VAT ID, or two related short notices) via a wrapping `<div style="display:flex; gap:24px">` containing a `<div>` per column — the parser already detects `display:flex` and maps it to a real flex container, and an explicit `gap` on any wrapping `<div>` becomes real spacing (`flex_gap`) between its children, tighter and more intentional than relying on Elementor's per-container defaults.
    - This is about *making a deliberate layout decision*, not reproducing any specific column count/percentage/font choice — a real design a user built by hand for this exact scenario grouped 4 short blocks into a 3-column row, then grouped the remaining longer sections into a wrapping flex row that lets them wrap naturally; that's one reasonable shape, not a template to copy verbatim. Pick groupings that fit the actual content.
- **A specific widget is named** ("add an accordion FAQ", "add a countdown", "add a testimonial") that `page_createFromHtml` wouldn't produce on its own → `page_addElementorWidget`, after the target page exists (create it first with either tool above if it doesn't yet). See `ELEMENTOR_WIDGETS.md` in this skill for `widgetType`/`settings` values and how confident each one is — it's not all equally reliable, and some (Loop Grid, Portfolio, Off-Canvas) have real limitations spelled out there. Say so to the user instead of presenting a best-effort result as finished.
  - **Building a content-heavy page with several `page_addElementorWidget` calls on the same page: issue them one at a time, sequentially, never in parallel.** Each call reads the page's whole element tree and writes the whole thing back — overlapping calls on the same post can race and silently drop each other's elements, the same failure class as the Kit-settings rule in 3a. This is a confirmed real bug, not a theoretical one: a page built via 4 sequential `page_addElementorWidget` batches ended up completely empty, including content from the first batch that had briefly rendered correctly on its own. Prefer fewer, larger calls (one container with several children) over many small ones when the content is naturally grouped.
  - If a page looks empty or wrong after building it, call `page_getElementorData({ id })` to read back what's actually persisted in `_elementor_data` before assuming a specific step failed or reporting anything as done — `page_get`'s `content` field is always empty for an Elementor page by design (it's the classic WordPress `post_content` field, never what Elementor actually renders from), so it's useless for this check.
- **A typo or wrong value in an already-existing widget** (created via `page_addElementorWidget` or `page_createFromHtml`) → `page_updateElementorWidget({ id, elementId, settings })`, not a full rebuild. `settings` is shallow-merged into that element's existing settings, so pass only the keys that actually change. `elementId` is the element's own `id` — from `page_addElementorWidget`'s response when the widget is created, or read back anytime via `page_getElementorData`.

## 3a. Kit baseline setup — check once per site before the first page/template

Before the *first* `page_create`/`page_createFromHtml`/`page_addElementorWidget`/`elementorTemplate_create` call on a given site in this conversation, call `elementorKit_getLayout` once. If it comes back with no `container_width` set, this site's Kit has never been customized — apply the baseline below before building anything. If `container_width` is already set (from an earlier task, possibly in a different conversation — this is real, persisted site state, not session state), skip straight to building content; the baseline only ever needs doing once per site.

**Issue every `elementorKit_*`/`elementorSettings_*` write one at a time, sequentially — never fire several as parallel tool calls.** Every one of them reads the Kit's entire settings blob, changes its own bit, and writes the whole thing back — there's one storage row, not one per setting. Two overlapping writes silently drop each other's change (whichever response lands last wins), which is exactly how a Kit's colors/fonts can look correct one moment and be back to Elementor's stock defaults the next, with no error anywhere.

1. **Layout — always, regardless of whether a design was given:** `elementorKit_updateLayout` with `containerWidth: 1300` and all six breakpoints active (`activeBreakpoints: ["viewport_mobile","viewport_mobile_extra","viewport_tablet","viewport_tablet_extra","viewport_laptop","viewport_widescreen"]`, keeping Elementor's own default px values for each unless told otherwise). This is a basic setup step, not something tied to a specific design — do it even for a single-page request.
1a. **GDPR/DSGVO — always, mandatory, not a design choice:** `elementorSettings_update({ googleFontsEnabled: false, loadGoogleFontsLocally: true })`. This disables Elementor's own live-from-Google Fonts integration and, as a safety net, makes it self-host any Google Font that's still referenced somehow instead of loading it from Google's servers — do this even on a plain content request with no design at all, unless the user explicitly says not to. (This is a real Elementor setting, confirmed in its own source — not this project's own invention.)
2. **Colors/Fonts — only if the request includes or implies a brand/design spec** (a color palette, typography, a mockup, a screenshot): set up the Kit's Global Colors/Fonts before building any page — never hardcode the design's colors/fonts directly into page content when a reusable global palette makes more sense, and never as a "Snippet"/custom CSS — write via `elementorKit_*`, which edits the real active Kit's settings natively. If no brand/design spec exists yet, skip this step silently and build with Elementor's defaults — don't interrupt a plain content request to ask for a palette that was never part of it.
   - `elementorKit_listColors`/`elementorKit_listFonts` first, to see what's already there (a fresh site already has 4 system color slots — primary/secondary/text/accent — and matching system font slots).
   - Rename the system slots to match the design's own naming via `elementorKit_updateColor`/`elementorKit_updateFont` with a new `title` (e.g. "Primär" → "Überschrift H1", "Akzentfarbe" → "Hervorgehoben") — renaming a system slot's title is supported, only its `_id` is fixed.
   - Set each system slot's `color`/`typography` to the design's values.
   - For every other palette color/typeface the design defines, add it as a custom entry via `elementorKit_addColor`/`elementorKit_addFont` (there's no fixed limit). Always include a plain white and black among the custom colors if the design doesn't already define its own equivalents.
   - Check `elementorKit_listFonts`' `defaultGenericFonts` — this is a single Kit-wide setting (Elementor calls it "Fallback-Schriftfamilie" in its editor, real field `default_generic_fonts`, **not** a per-font `typography_*` field). If it's unset, call `elementorKit_updateFontFallback({ defaultGenericFonts: "Arial" })` once. If it's already set — from an earlier baseline run, or the user set it manually in the Elementor editor — leave it alone; only change it again if the user explicitly asks for a different fallback.
   - `typography_font_family` only actually renders if that typeface is available. **Never reference a Google Font by name directly** — Elementor/the browser would load it live from Google's CDN, which is a GDPR/DSGVO problem for an EU-facing site (a well-known real legal risk, not a style preference). Instead, self-host every font — including Google Fonts — via `customFont_upload` (see section 5h): download the font file first, then upload it, then reference that same `family` name. A system font (Arial, Helvetica, ...) is the only case that needs no upload at all.

   Every page/template built afterward with `page_createFromHtml`/`elementorTemplate_create` can then reference these same global tokens instead of hardcoding colors per page.

## 3b. Building a whole site from a design ("build me a site based on this template")

Break this into separate tool calls per piece — never fold the footer or navigation into a page's own content. Run the 3a baseline check first, then:

1. **Each page** (home, about, contact, ...) → `page_createFromHtml` (see section 3), one call per page.
2. **Footer and header** → `elementorTemplate_create` with `type: "footer"`/`"header"` (see section 5) — **not** a section added to a page's own content. A footer built as page content only appears on that one page; a footer built as a Theme Builder template appears site-wide and shows up correctly in Elementor's own "My Templates" screen. Check `elementorTemplate_list({ type: "footer" })` first (or whatever type is relevant) before creating one — it lists every template of that type with its `lang`, so an already-existing footer/404/search-results/... template (possibly hand-built in the Elementor editor, with no language tag at all) doesn't get silently duplicated. After creating one, tell the user to spot-check it in `/wp-admin/edit.php?post_type=elementor_library` or the Elementor editor.
3. **Navigation menu** → `menu_create` (name only, creates an empty menu), then `menuItem_create` per link (see section 5g) — **not** a page or Theme Builder header. Assigning the finished menu to a theme location (header/footer nav) has no API; tell the user to do that manually in WP-admin → Appearance → Menus after building it.

Report each piece's outcome separately (page IDs, footer/header template IDs, menu ID) rather than one blanket "site built" summary — that's what makes a partial failure (e.g. the footer template silently not saving correctly) visible instead of hidden inside an overall success claim.

## 4. Publishing

`page_createFromHtml` always leaves the resulting page as a **draft**, even after the conversion job completes — publishing is a separate step. Once the job is `"completed"`, call `page_update({ id: postId, status: "publish" })` unless the user explicitly asked to leave it as a draft for review first.

## 4a. Multilingual sites (Polylang or WPML)

`site_healthCheck`'s result includes a `languages` field — `null` on a single-language site, otherwise `{ plugin, default, available }`. If it's set and the user asks to create or update content without saying which language it's for, **ask before calling `page_create`/`page_createFromHtml`/`page_update`/`elementorTemplate_create`** — don't silently assume the site's default language. Pass the answer as `lang`. If the new page/template is meant to be a translation of an existing one, ask for that page's/template's ID too and pass it as `translationOfId` to link them as a translation group.

**This applies to Theme Builder templates too, not just pages — don't rely on a language-suffixed title alone.** Naming a template `"Footer DE"` (see section 5) is purely cosmetic; without also passing `elementorTemplate_create`'s `lang`, Polylang assigns whatever its own site default is (often English), regardless of what the title says or what language the content is actually written in. If a template already exists with the wrong language (created before `lang` was passed, or before this rule existed), fix it with `elementorTemplate_updateLanguage({ id, lang, translationOfId? })` — there's no general "update a template" endpoint, so this is its own tool rather than folded into something else.

**Title stays in the admin's own working language; the slug gets properly translated.** These are two independent axes: the *admin's own working language* (the one they read/write, ask if unclear — never assume it equals the site's Polylang `default`, confirmed live: a real site had Polylang `default: "en"` while the admin's actual working language, and the language they build content in first, was German) and the *site's frontend language(s)* (whatever `available` lists). When translating an existing page/template into another frontend language:
  - **Title**: keep the same admin-language wording across every language variant, only the language suffix changes — e.g. `"Impressum DE"` → `"Impressum EN"`, never `"Legal Notice EN"`. If the title were translated into a language the admin doesn't read, they'd lose the ability to recognize their own content in the admin list.
  - **Slug/URL** (`page_create`/`page_createFromHtml`'s `slug`, where applicable): must be the real, correct native term in the *target* language, not a transliteration of the admin-language title — e.g. German `Impressum` (slug `impressum`) translates to English `Imprint` (slug `imprint`), not `impressum`. Get the slug from the actual translated content, not by sanitizing the admin-language title.
  - **Everything else** (body text, image alt text, button labels, meta fields) — fully translated into the target language, same as normal.
  - This is a general principle, not specific to any one feature: an American building a French site keeps admin-language (English) titles suffixed per language, but writes proper French slugs — same rule regardless of which language pair is involved.

## 5. Theme Builder templates — footer, header, 404, search-results, single-post, product, popup

A request for a site-wide **footer**, **header**, custom **404 page**, **search-results page**, **single-post/single-[custom-post-type] template**, WooCommerce **product template**, or **popup** → `elementorTemplate_create({ title, type, elements, conditions?, pageSettings?, lang?, translationOfId?, siteId })`. Build `elements` the same way as `page_addElementorWidget`'s `element` (see `ELEMENTOR_WIDGETS.md`), just as a top-level array — for single-post/product templates, use the context-only widgets documented there (`theme-post-title`, `theme-post-content`, `woocommerce-*`, ...).

- `type`: `"footer"`, `"header"`, `"error-404"`, `"search-results"`, `"single-post"`, `"product"`, or `"popup"`.
- `title`: follow Elementor's own naming convention exactly — the display name a user would type into "Add New Template" (e.g. `"Footer"`, `"Header"`). On a multilingual site (see section 4a), append the language, e.g. `"Footer DE"`/`"Footer EN"` — one template per language, each with its own `conditions` scoped to that language (don't rely on `type` alone to disambiguate them) **and its own `lang` actually set** — the title suffix is cosmetic, `lang` is what Polylang/WPML actually use.
- `conditions`: only `footer`/`header` default to `["include/general"]` (entire site) if omitted. **Every other type requires an explicit condition** — see `elementor-templates.md`'s suggested-strings table (e.g. `["include/404"]`, `["include/singular/post-type/product"]`). Ask the user if it's unclear what the condition should be rather than guessing. None of these condition strings are verified against a live install — say so, and check the result in the Elementor editor.
- `pageSettings`: mainly for popups (width, close-button, background). **After creating a popup, always tell the user its trigger (on page load, on scroll, on exit intent, ...) still needs setting manually in the Elementor editor's Popup Settings → Triggers tab** — there's no ground truth for that config yet, so this tool can't set it, and an untriggered popup never shows to visitors.
- **Content: always widget-by-widget with the matching native Elementor widget** — a heading is a `heading` widget, a paragraph a `text-editor` widget, a list an `icon-list` widget, a social row `social-icons`, a CTA a `button`, and so on (see `ELEMENTOR_WIDGETS.md` for the full catalogue and each widget's settings shape). Never dump the whole footer/header as one `html` widget for convenience — that's a last-resort exception only for content with no matching native widget at all (e.g. a raw embed script), not a default shortcut. Widget-by-widget is what makes the result editable in the Elementor UI afterwards the way a manually-built template would be.
- **Fixing a template's content afterward** (a typo, a wrong link, a layout tweak) — `page_addElementorWidget`/`page_updateElementorWidget` work on a template's post ID exactly like a page's, since there's no template-specific equivalent. The template's real type (`footer`, `header`, ...) is now preserved automatically when doing this (confirmed real incident: it previously silently reverted to a plain page, un-registering the template from Elementor's Theme Builder entirely — fixed in `veilwright-ai@0.18.1`) — but this only applies once that update is installed; before it, any such "fix" call risked breaking the template. If a template ever silently stops showing up as its type in Elementor's Templates list after being edited this way, that's this exact failure mode — check the plugin version.
- **Deleting a template** — `elementorTemplate_delete({ id })`, not `page_delete` (that's scoped to regular pages, 404s on a template's post — a template is `elementor_library`, not `page`). Use this to clean up after replacing a wrongly-configured template (wrong type, wrong language, ...) with a corrected one, rather than leaving the broken one sitting around.
- **Layout, not just a vertical stack** — `elements` here is raw Elementor JSON, not converted from HTML, so none of the HTML-authoring conveniences apply; the layout is *entirely* what's in each container's own `settings`. A flat list of top-level containers with no explicit layout settings renders as one column, full-width, evenly spaced by Elementor's defaults — rarely what's wanted for a footer with several columns.
  - **For a multi-column row (a footer's logo/contact/legal columns, card/step rows, etc.), use plain flex with percentage child widths — see `ELEMENTOR_WIDGETS.md`'s "Multi-column row" section for the full, live-verified copy-paste pattern (checked at desktop/tablet/mobile plus a hover state, 2026-07-14).** Superseded guidance, corrected: an earlier test (2026-07-10) concluded neither `width: { "unit": "%" }` nor `flex_grow` worked on a flex child and recommended Container Grid mode instead — that grid recommendation turned out to have its own real bug (a grid container silently collapses to one column unless `content_width: "full"` is set, confirmed 2026-07-15 by reading Elementor's generated CSS directly) and, separately, comparing against a real hand-built reference showed **percentage width on a flex child does work** — the 2026-07-10 test was very likely missing that same `content_width: "full"` setting on the child, which is required either way. Bottom line: `flex_direction: "row"` + `flex_wrap: "wrap"` on the parent, `width: { "unit": "%", "size": N }` + `content_width: "full"` on each child — no grid mode needed for a same-size-boxes-in-a-row layout. **This is the verified mechanism, not a fixed template** — exact widths/gaps/colors/breakpoints/column counts always come from the actual design or instructions given for that specific page, never copied wholesale from a reference example. `ELEMENTOR_WIDGETS.md`'s "Multi-column row" section also covers a featured-card variant (differentiated border, `flex_justify_content:"space-between"` for aligned buttons) and an overlapping badge/ribbon technique via Elementor's Advanced-tab Position controls (`_position:"absolute"`, `_offset_x_end`/`_offset_y`) — same idea: real, reusable mechanisms, applied per the actual design at hand.
  - **Give every container a short, meaningful `_title`** (e.g. `"Preise"`, `"Spalten"`, `"Single Site"`, not left unset) — purely cosmetic (shows in the Elementor editor's Structure/Navigator panel only), but it's what lets the page owner find the right container by hand afterward instead of clicking through nameless ones.
  - Within a single column (stacking a heading above its paragraph, vertically), plain `flex_direction: "column"` + `flex_gap` is fine and confirmed working — the child-width problem is specific to *side-by-side* flex layout, not flex/column layout generally.
  - If a setting looks right in `page_getElementorData` but the live page doesn't reflect it, that's not proof it's applied — Elementor's per-element CSS is a separate generation step; check computed styles in a real browser (or ask the user to look) rather than trusting the stored JSON alone for anything layout-related that hasn't been separately verified.
- **Images need real accessible text.** The `image` widget's own `alt` setting (`settings.image.alt`) is supported and should always be set for a meaningful image (a logo, a photo) — never leave it empty. When Elementor Pro is installed, also use its Advanced tab → Attributes panel for a `title`/`aria-label` where `alt` alone doesn't cover it (e.g. a logo that's also a clickable link) — real setting key confirmed live: `_attributes`, a single string, `key|value` per line (see `ELEMENTOR_WIDGETS.md`'s "Custom HTML attributes" section for the verified shape and an example). Don't skip this just because it's Pro-only — use it whenever Pro is installed, which is the common case for a real client site.
- **Links: prefer relative, slug-based URLs over parametric ones.** An internal link should be `/de/impressum`, not `https://veilwright.one/de/impressum` or `/?page_id=233` — the latter breaks if the site ever moves domains and isn't stable if the post is ever recreated. If only a page ID is known, resolve its real slug via `page_get` first rather than emitting a `?page_id=` URL.
- **Choosing how to link**: a bare link inside a `text-editor` widget's HTML is the least accessible/flexible option (no independent styling, easy to miss visually) — prefer it only for genuinely inline, sentence-level links (e.g. "see the [privacy policy](...)" inside a paragraph). For a standalone link (a footer "Impressum" link, a CTA) use a `button` widget instead — it can be styled with a transparent background, borders, hover state, etc., and reads clearly as an interactive element for screen readers. For a whole navigation list (a footer link column, a header menu) prefer a real WordPress menu (`menu_create`/`menuItem_create`, see section 5g) referenced via Elementor's native Nav Menu widget over hand-building individual link widgets — but that widget's exact `widgetType`/settings shape isn't verified yet here, ask the user to confirm it renders correctly in the editor after using it, same caveat as any other unverified widget.
- **A running/dynamic year in copyright text** (`© 2026 ...`) — never hardcode the year, it goes stale. Check `codeSnippet_list` for an existing shortcode first (this project's convention: a `[y]` shortcode returning `date('Y')`, sometimes paired with `[c]` for `&copy;`) before assuming one needs creating — write `[y]` (or whatever the site's existing snippet defines) into the `text-editor`/`heading` widget's text instead of a literal year; Elementor renders shortcodes in text widgets through `do_shortcode()`.
- **Every footer needs breadcrumbs, and a multilingual site's footer needs a language switcher.** For breadcrumbs, use Elementor Pro's **native `breadcrumbs` widget** (see `ELEMENTOR_WIDGETS.md`) — it's Elementor's own breadcrumb generator, independent of whichever SEO plugin is active, and gets full Elementor style controls. **Do not build a custom `[yoast_breadcrumb]`-style shortcode for this** — confirmed live (2026-07-10) that the native widget is both simpler and strictly better (a shortcode-in-text-editor approach was tried first, then found to be an unnecessary reinvention once a real footer using the native widget was seen). For the language switcher on a multilingual site (section 4a), same story: Elementor Pro ships a **native `polylang-language-switcher` widget** when Polylang is active — use that, not a `[language_switcher]`-style shortcode, same reasoning. Style both to match the site's existing footer design (colors via Kit `__globals__`, typography settings) rather than copying a fixed layout — the right placement/styling depends on what the rest of the footer already looks like.

A request for anything else in Elementor's Theme Builder catalogue — Loop Item templates, Cookie Consent, Preferences Banner — is **still not implemented**, there's no tool for it. Say so plainly rather than approximating one by editing a single page's content, which would not behave like a real site-wide/conditional template and would mislead the user about what happened. See `ELEMENTOR_WIDGETS.md`'s "Theme Builder templates" section for the full list of what's been confirmed to fall in this bucket.

### Third-party plugin widgets (Fluent Forms, Smart Slider 3, JetSmartFilters, ...)

These belong to plugins other than Elementor/Elementor Pro — there's essentially no visibility into their settings schema, so don't guess a `widgetType` for them (a wrong slug silently renders nothing).

**Fluent Forms and Smart Slider 3 are verified exceptions** (see `ELEMENTOR_WIDGETS.md`): if the user wants an *existing* form/slider embedded and can give you its ID (Fluent Forms → All Forms, or Smart Slider 3, in WP-admin), use `fluent-form-widget`/`smartslider` directly — that part is well-defined.

For a **brand-new Fluent Form**, use the `form_createFluentForm` tool to actually build it (fields, submit button, settings, notifications — see `FLUENT_FORMS.md` in this skill for the shape), then embed the resulting form ID with `fluent-form-widget`. Only fall through to the placeholder procedure below if the fields needed aren't in `FLUENT_FORMS.md`'s known-`element` table (payment fields, multi-step, conditional logic aren't verified yet) — say so and ask, don't guess an unverified field type. Smart Slider 3 and everything else still has no creation path — that case always falls through to the steps below.

Before falling back to a placeholder for anything else, check whether the plugin has a plain WordPress **shortcode** for it (a lot of older plugins do, even without a dedicated Elementor widget) — a `shortcode` widget (`{ "widgetType": "shortcode", "settings": { "shortcode": "[...]" } }`) can embed that directly, see `ELEMENTOR_WIDGETS.md`.

If there's no shortcode either:

1. Add a plain placeholder where it belongs instead — a `heading` or `text-editor` widget saying something like "TODO: [Fluent Forms contact form] — add manually in the Elementor editor" (adjust wording to what's actually needed), via `page_addElementorWidget`.
2. Tell the user explicitly: this widget type isn't supported, a placeholder was left at the right spot, and ask whether they'd rather (a) build it themselves in the Elementor editor, or (b) have Claude add a *generic* Elementor equivalent instead where one reasonably exists (e.g. Elementor Pro's own `form` widget — see `ELEMENTOR_WIDGETS.md`'s Tier C).

Don't silently pick option (b) without asking — a generic substitute isn't necessarily what the user's site actually needs (e.g. if a form's submissions feed an existing integration, a different form widget breaks that).

## 5b. Custom code snippets (Code Snippets / codesnippets.pro)

**Before reaching for a site-wide `site-css`/`admin-css` snippet, check whether the styling is actually scoped to one specific element or a handful of related elements — if so, use Elementor's own per-element Advanced-tab fields instead (`custom_css`/`css_classes`/`_element_id`, see `ELEMENTOR_WIDGETS.md`'s "Per-element custom CSS/ID/classes" section), not a Code Snippet.** A Code Snippet CSS snippet is for genuinely site-wide styling that isn't tied to specific elements Claude is building; anything scoped to a particular widget/section belongs on that widget, the same way a specific widget setting is always preferred over custom CSS in the first place.

For custom PHP/CSS/JS/HTML — not Elementor content — use `codeSnippet_create`/`codeSnippet_update`/`codeSnippet_list`/`codeSnippet_get`/`codeSnippet_delete`/`codeSnippet_activate`/`codeSnippet_deactivate`, not a page tool. Valid `scope` values: `global`, `admin`, `front-end`, `single-use`, `content`, `head-content`, `footer-content`, `admin-css`, `site-css`, `site-head-js`, `site-footer-js`, `condition` — the CSS/JS/content ones only actually execute with Code Snippets **Pro** active, though the free plugin still stores them. A new snippet is created inactive unless `active: true` is passed, and PHP snippets with a syntax error are forced back to inactive by the plugin itself — check the returned `codeError` field and tell the user if it's set rather than reporting success. `condition_id` (Pro's snippet-conditions feature) isn't supported.

## 5c. SEO metadata (Yoast, Rank Math, All in One SEO, SeoPress)

Per-plugin tools, not one generic one: `yoast_get`/`yoast_update`, `rankmath_get`/`rankmath_update`, `aioseo_get`/`aioseo_update`, `seopress_get`/`seopress_update` — all share the same field shape (`title`, `metaDescription`, `focusKeyword`, `canonical`, `noindex`, `nofollow`, `ogTitle`, `ogDescription`, `ogImage`, `twitterTitle`, `twitterDescription`, `twitterImage`). Check `site_healthCheck`'s `adapters` list for which one is actually active before calling any of them — calling the wrong plugin's tool 424s (dependency missing), it doesn't silently no-op.

## 5d. WooCommerce

`woocommerceProduct_list`/`_get`/`_create`/`_update`/`_delete` — simple products only. Variable products (with variations/attributes) aren't supported — if the user needs those, say so rather than creating a simple product that doesn't match what they asked for.

## 5e. Custom field plugins (ACF, Pods, Meta Box)

`acf_getAll`/`acf_get`/`acf_update`/`acf_updateAll` (ACF), `pods_get`/`pods_update`/`pods_updateAll` (Pods), `metaBox_get`/`metaBox_update`/`metaBox_updateAll` (Meta Box) — `selector` is the field name/key. Check `site_healthCheck`'s `adapters` list for which plugin is actually active before calling any of them. Pods and Meta Box have no "get all fields" tool (no stable API for that) — ask the user for the specific field name/key rather than guessing one.

## 5f. Form plugins with dedicated tools (Contact Form 7, WPForms, Ninja Forms, Formidable Forms)

For these four, use each plugin's own tools instead of an Elementor widget or a page tool: `cf7_list`/`_get`/`_create`/`_update`/`_delete` (Contact Form 7), `wpforms_list`/`_get`/`_create`/`_update`/`_delete` (WPForms), `ninjaForms_list`/`_get`/`_create`/`_update`/`_delete` (Ninja Forms), `formidable_list`/`_get`/`_create`/`_update`/`_delete` (Formidable Forms). Check `site_healthCheck`'s `adapters` list first for which one is actually active — calling the wrong plugin's tool 424s.

Each plugin's `fields`/`settings`/`mail`/`options` are passed through verbatim in that plugin's own shape — there's no shared/generic form-field schema across them:
- **Contact Form 7**: `form` is CF7's own shortcode-tag markup (e.g. `[text* your-name][submit "Submit"]`), not HTML. `mail`/`mail2` use CF7's field names (`subject`, `sender`, `body`, `recipient`, `additional_headers`, `attachments`, `use_html`). `messages` is a flat map of message keys (e.g. `mail_sent_ok`) to text.
- **WPForms**: `fields`/`settings` use WPForms's own JSON shape, the same shape you'd see in an exported form. Creating without `fields` produces an empty form. `_update`/`_delete` require the WP user mapped to the API key to hold WPForms's own capabilities — if a call fails with no detail, that's the likely cause.
- **Ninja Forms**: `fields`/`actions` use Ninja Forms's own settings-object shape. A default "Save Form" action is added automatically if `actions` is omitted on create. `ninjaForms_update` only changes the form's `title`/`settings` — fields/actions can't be edited after creation with this tool; recreate the form if they need to change.
- **Formidable Forms**: each field object has `type`/`name`/`fieldOptions` and, for choice-based types, `choices`. `options` is the form-level options object (before/after/submit HTML, etc.). `formidable_delete` trashes the form (recoverable) unless `permanent: true` is passed, which also removes its fields/entries/actions.

None of these four plugins' form-builder UIs were used to hand-verify a real created form's on-site rendering — say so if the user asks for confirmation beyond the API response.

## 5g. Navigation menus

`menu_list`/`_get`/`_create`/`_update`/`_delete` for the menu itself; `menuItem_list`/`_create`/`_update`/`_delete` for its items.

- A menu item is either a **custom link** (set `url`) or a **link to an existing page/post/category** (set `objectType` — e.g. `"page"`, `"post"`, `"category"` — and `objectId` to its ID; WordPress derives the real URL itself). Don't set both.
- `parentId` (another item's ID) nests an item as a submenu entry; `position` controls display order.
- There's no API for assigning a menu to a theme location (the actual header/footer nav slot) — after building the menu, tell the user it still needs assigning manually in WP-admin → Appearance → Menus → Manage Locations.
- `menu_create` only takes a `name` — it starts empty; add items afterward with `menuItem_create`.
- **Always set `titleAttribute` when creating a menu item** — `menuItem_create` requires it. This is the HTML `title=""` tooltip attribute WordPress's own classic menu editor calls "Title Attribute" (real WordPress core field, confirmed against `wp-includes/nav-menu.php`'s `menu-item-attr-title` arg — stored on the menu item post's own `post_excerpt`, distinct from `title`, the visible link label). Write it **SEO-optimized** — a short, natural, genuinely descriptive phrase about the destination, not just the label text repeated — and **in the menu's actual language** (check the site's `languages`/the page's `lang`, don't assume from the label alone on a multilingual site, same as rule 4a). Fixing an existing item that's missing one → `menuItem_update({ ..., titleAttribute: "..." })`, check via `menuItem_list` first.

## 5h. Custom (proprietary/self-hosted) fonts

`customFont_upload(filePath, family, weight?, style?)` uploads a local `.ttf`/`.otf`/`.woff`/`.woff2`/`.eot` file and registers a real `@font-face` for it, printed on every page load (frontend and Elementor's editor preview both hit it) — **not** via Elementor Pro's own Custom Fonts screen (that data model is closed-source, no API), but functionally equivalent from the browser's side: once uploaded, reference the same `family` name in `elementorKit_addFont`/`updateFont`'s `typography_font_family` and it renders correctly. It will **not** appear in Elementor's own Custom Fonts admin list/picker — that's cosmetic only, doesn't affect rendering.

- One upload per weight/style combination — call it again with the same `family` for a bold/italic variant.
- `customFont_list`/`customFont_delete` to see/remove what's registered.
- **Use this for every font, including Google Fonts** — never reference a Google Font by name/CDN directly (GDPR/DSGVO: the browser would load it live from Google's servers). Download the font file first, then upload it here, so it's fully self-hosted.
- **Elementor Pro's own Custom Fonts screen** (WP-admin → Elementor → Editor → Custom Elements → Fonts) is a separate, Pro-only capability — only available if the user has Elementor Pro installed, and something they'd do manually there (real TTF/WOFF upload through Elementor's own UI). A font uploaded that way *does* show up in Elementor's font-family picker dropdown, unlike one registered via `customFont_upload`. Both approaches make the font render correctly; only the admin-UI-visibility differs. If the user already has fonts set up that way (check `elementorKit_listFonts`'s `typography_font_family` values against what's visible in their Custom Fonts list), don't re-upload duplicates via `customFont_upload`.

## 5i. Blog posts & category-driven feeds (changelog, news, ...)

`post_create`/`post_get`/`post_update`/`post_delete`/`post_list` — the built-in WordPress **Post** type (`post_type: "post"`), not a page and not an Elementor template. Same shape as the `page_*` tools (`title`/`content`/`status`/`lang`/`translationOfId`), plus `categories`: an array of category names (or numeric term IDs) — **a name that doesn't exist yet on the site is created automatically**, no separate lookup/creation step needed. `post_list`'s optional `category` filters to one category (exact name match) — use it to check what's already published before creating more (e.g. "does a changelog post for this version already exist in this language").

**Use this for anything that should render as an automatically-updating, chronological, categorized feed** rather than content you'd hand-maintain on a fixed page — a changelog, a blog, news/updates. The pattern: tag each entry with a category (e.g. `"Changelog"`), then build **one** page per language with a native `posts` widget filtered to that category (see `ELEMENTOR_WIDGETS.md`'s Posts widget entry) — WordPress sorts by publish date automatically, so a new post just appears, no page edit required. Multiple categories on one post work too (e.g. `["Changelog", "Plugin"]` vs `["Changelog", "MCP-Server"]`) for a sub-grouping/badge without a second taxonomy.
